Classes vs Modules (Technical blog) 10/24/2014

In my previous technical blog post, I described what a Ruby class is, and what it's good for. Here, I will compare and constrast it to Ruby modules.

As a refresher, a Class in Ruby is like a blueprint for building an object. Remember, everything in Ruby is an object, and classes are a way for us to create an object and reuse it over and over again without repeating the code. Also, remember that in Ruby, classes can only experience single-inheritence. That is, a class can only inherit from one other class, and they cannot be chained together.

Like classes, a Module allows us to reduce repetition in our code. Although it is technically an object, it's more helpful to think of it as "a part" that can be used for other objects (like classes). For example, if we had class Automobile and class Wheelbarrow, we could create modules to represent parts that both share (at least in the most basic design principle):

module Wheel

def move

-----puts "Rolls forward"

end

end

We could then use this module within our automobile and wheelbarrow classes:

class Automobile

include Wheel

end

class Wheelbarrow

include Wheel

end

Tesla_roadster = Automobile.new
Tesla_roadster.move #=> Rolls forward

Our Wheel module allows us to define what a wheel is and use it in any class object that contains a wheel, without having to re-code the wheel over and over in a code! This would be messy and not really possible if Wheel was a class, and we wanted our Automobile class to inherit other pieces and parts.
